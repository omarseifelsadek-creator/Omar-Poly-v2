## Critical findings

### C1 — Fill bookkeeping is dead code [verified]
`execution/pair_runner.py:776-897`. Inside `_try_evaluate()`, the rejection branch
(`if not action or action.get("rejected"):`) ends with `return` at line 776 — but ~120 lines of
success-path bookkeeping sit *after* that return, **at the same indentation level inside the
rejection branch**. Unreachable on every path:

- `_live_fills` append (live P&L tracking) → live P&L falls back to paper-engine numbers (**wrong while trading live**)
- Zone counts, hedge times, capital-exhaustion time, max unhedged exposure, slippage tracking
- `log_pair_buy(...)` CSV logging → `pair_buys_*.csv` gets no rows
- Dashboard recent-buys display, hourly report fills, console BUY output

Secondary bug inside the dead block: line ~797 zone classification `else: zone = "Panic"` should be
`"Dead"` (the rejection-path copy at line 760 is correct).

Likely cause: indentation regression when the rejection-handling block was added.
**Impact: every paper/live session since the regression produced no fill data; live P&L misreported.**

### C2 — Trades recorded on hardcoded BUY side [verified]
`main.py:187-192`. `level_tracker.record_trade_at_level(price=..., side=Side.BUY, ...)` — the level
side is hardcoded regardless of trade direction. Per the convention documented at
`state/level_tracker.py:274-276`, a BUY trade (taker lifted the ask) should record on the ASK side,
a SELL on the BID side. Corrupts absorption detection (Phase 2 analytics) in the intelligence-dashboard path.

### C3 — Critical dependencies missing from requirements.txt [verified]
- `py-clob-client` — imported at `execution/executor.py:64-67`; live/dry-run modes crash on fresh install.
- `openpyxl` — imported by `generate_pair_report.py` and `generate_report.py`; report generation crashes.
- `sklearn` — imported by `quant_dashboard.py` only (superseded script; retired to research/legacy/ rather than adding the dep).

### C4 — Unbounded module-level dedup dicts (memory leak)
`analytics/interpreter.py:35` (`_last_emitted`) and `analytics/signals.py:59` (`_last_signals`).
Keys include price levels (e.g. `wall_{price}_{side}`), so churning markets mint new keys forever.
No eviction → steady memory growth on long-running live sessions.

### C5 — Live mode has no startup validation or confirmation
- `execution/executor.py:177` — `LiveExecutor` loads `.env` lazily; missing/malformed credentials
  surface only at the *first order attempt*, mid-window.
- No confirmation gate: `--mode live` starts placing real-money FOK orders with no prompt.

## What's healthy (verified where marked)

- **Live order path is real and correct**: py-clob-client EIP-712 signing, FOK orders,
  `signature_type=1` (Magic L2 proxy) + `POLY_FUNDER` both set (omitting either → "not approved
  operator"), lazy client init with TLS warm-up, engine-state snapshot/rollback on rejection.
- **Fee model correct**: `price * (1 - price) * 0.0625` applied in both engines.
- **Analytics fully wired**: spoofing/absorption/sweep detectors, CVD (+divergence), momentum/regime,
  liquidity voids — all flow into Metrics → insights → signals. No orphaned phases.
- **`ui/cyber_engine.py` + `ui/cyber_dashboard.py` are NOT orphaned** [verified] — default no-flag
  mode via `main.py:1049` (an audit agent initially misreported this).
- WebSocket reconnect w/ exponential backoff; bounded DB write queue w/ batch flush; settlement
  fallback chain Chainlink → Binance → Gamma → order book; comprehensive `.gitignore`; clean git tree.
- Pairs engine logic (zones, skew locks, atomic entry, panic hedge, theta sizing, VWAP book-walking,
  time-in-book fill model) — sophisticated and internally consistent; v15 + Gemini patches integrated.
